# Redline

[![ci](https://github.com/Kasamix/redline-llm/actions/workflows/ci.yml/badge.svg)](https://github.com/Kasamix/redline-llm/actions/workflows/ci.yml)

A from-scratch LLM inference engine in C++/CUDA: paged KV cache, iteration-level continuous batching, CUDA-graph decode, custom GQA attention kernels. Serves Qwen2.5-1.5B-Instruct in FP16 on a single consumer GPU.

**Headline** (RTX 4090, median of 3 trials, [measurement contract](bench/FAIRNESS.md)): **4,748 output tok/s** on the 64-stream primary workload - ahead of vLLM 0.9.2 (4,420), at 91% of vLLM 0.24.0 (5,195), and 3.6× llama.cpp (1,320) - with byte-identical greedy outputs verified across all engines before any measurement, and the losing cases published alongside the wins.

**Status: measured results published.** The benchmark section below is generated by `bench/report.py` from the committed results JSON under `bench/results/` and is guarded by CI against manual edits; the headline paragraph above restates that section's medians (primary-case output throughput) and contains no number that does not appear in the generated tables. Every published number follows the reviewer-facing measurement contract in [bench/FAIRNESS.md](bench/FAIRNESS.md), including the cases where redline loses.

## Architecture

- Custom CUDA kernels: fused RMSNorm + residual, rotary embeddings, paged GQA attention (online softmax, FP32 accumulation), KV scatter, fused SiLU-mul; cuBLASLt for GEMMs
- Paged KV cache (16-token blocks, block-table indirection) - concept from the PagedAttention paper (Kwon et al., SOSP 2023)
- Iteration-level continuous batching scheduler with chunked, composite prefill
- CUDA graph capture for decode steps across batch-size buckets
- `safetensors` mmap weight loading; tokenization handled client-side (the engine consumes and emits token IDs)

Full design rationale: [docs/DESIGN.md](docs/DESIGN.md). Measured profiling (kernel-time breakdown, timeline gap analysis, DRAM-traffic bound, prefill accounting): [docs/PROFILING.md](docs/PROFILING.md).

## Build

**Linux (including WSL2) only.** The build assumes a POSIX toolchain - GCC/Clang with a CUDA 12.6+ toolkit - and passes GCC-style warning flags unconditionally; Windows-native (MSVC) builds are not supported (develop under WSL2 instead, which is exactly the `wsl-*` preset setup). Requires CMake with presets support and Ninja. Presets live in `CMakePresets.json`:

```
cmake --preset pod-release          # release build for the benchmark GPU preset
cmake --build --preset pod-release
ctest --preset pod-release          # CPU suites + GPU suites when a device is present
```

`wsl-debug` / `wsl-release` are the small-GPU development presets. The build produces the `redline` Python module (pybind11) and the `redline_bench` profiling CLI.

## Quickstart

```
python scripts/fetch_model.py       # pinned Qwen2.5-1.5B-Instruct checkpoint (docs/MODEL_SPEC.md)
PYTHONPATH=build/pod-release python
```

```python
import redline

engine = redline.Engine("models/Qwen2.5-1.5B-Instruct")
request_id = engine.add_request(
    token_ids=[1000 + i for i in range(32)],  # the engine speaks token IDs, not text
    max_new_tokens=8,
    ignore_eos=True,
)
finished = False
while not finished:
    for req_id, token_id, done, reason in engine.step():
        print(req_id, token_id, reason)
        finished |= done
```

The API surface (pump loop, `has_capacity()` backpressure, `stats()`) is documented in `docs/DESIGN.md` section 10. The benchmark harness under `bench/` drives this same module in-process; reproduction steps are part of the generated section below.
